1. `time.sleep(3)`과 `await asyncio.sleep(3)`의 핵심 차이는 무엇입니까? 대기중 아무것도 안하는것과 다른 작업을 하고 와서 깨어주는것.
2. 모델 추론처럼 CPU를 계속 사용하는 작업에서 `async/await`만으로 동시 처리가 안 되는 이유는 무엇입니까? async/await는 주로 기다리는 시간이 많은 작업에 효과적. 실제로 cpu/gpu가 리소스를 사용해야 하는 일에는 쓰레드풀로 넘겨주는 방법을 사용해야지 벙렬적으로 처리가 가능하다.

1. `async def` 안에서 `time.sleep(3)`을 호출하면 왜 다른 요청까지 지연됩니까? 동기작업을 비동기 정의안에서 처리할때는 모델 추론이 끝날 때까지 이벤트 루프를 붙잡기 때문에 다른 요청까지 지연됩니다.
2. 일반 `def`로 선언된 엔드포인트는 FastAPI가 내부적으로 어떻게 처리합니까? 외부 thread pool에서 실행한 뒤 그 결과를 await하는 방식으로 처리합니다.
3. blocking 추론 중 헬스체크까지 막히면 실무에서 어떤 문제가 발생할 수 있습니까? 헬스체크 요청까지 막히면 서버가 실제로는 죽지 않았는데도 로드밸런서, Kubernetes, 클라우드 플랫폼이 서버를 비정상 상태로 판단할 수 있습니다.

1. `run_in_executor`가 이벤트 루프 블로킹을 방지하는 원리는 무엇입니까? 동기함수를 별도의 쓰레드풀에서 실행시키고 그 결과는 비동기적으로 기다린다. blocking 작업은 다른 스레드에서 돌리고, async 쪽에서는 결과만 await하는 구조
2. `run_in_executor`의 첫 번째 인자에 `None`을 넣으면 어떤 스레드풀이 사용됩니까? 기본 쓰레드풀
3. 일반 `def`와 `async def + run_in_executor`의 핵심 차이는 무엇입니까? 커스텀한 쓰레드풀 사용이 가능해 짐
4. GPU 추론 시 스레드풀 크기를 1~2로 제한하는 이유는 무엇입니까? GPU 추론은 동시 요청을 많이 넣는다고 선형으로 빨라지지 않는다. 오히려 GPU 메모리와 연산 자원을 두고 경쟁해서 OOM, latency 증가, 처리량 저하가 발생할 수 있다. 그래서 추론용 thread pool 크기를 1~2 정도로 제한해 GPU에 들어가는 작업 수를 통제한다.

1. 글로벌 Exception Handler를 사용하면 어떤 반복을 줄일 수 있습니까? 공통 예외 처리 로직을 한 곳에서 관리하여 에러 응답 형식 통일, 상태 코드 처리, 로깅, 민감 정보 숨김 같은 반복을 줄일 수 있습니다
2. 클라이언트에게 스택 트레이스를 노출하면 안 되는 이유는 무엇입니까? 공격자가 서버 구조를 파악하기 쉬워집니다. 사용자 입장에서도 긴 에러 로그를 받아봤자 이해하기 어렵고, 서비스 신뢰도도 떨어집니다.
그래서 클라이언트에게는 짧고 안전한 메시지만 보여주는 것이 좋습니다.
3. `logging` 모듈이 `print()`보다 나은 점은 무엇입니까?logging은 운영 환경에서 필요한 기능을 제공합니다. | 항목       | `print()` | `logging`                                       |
| -------- | --------- | ----------------------------------------------- |
| 로그 레벨    | 없음        | `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL` |
| 출력 위치    | 주로 콘솔     | 콘솔, 파일, 외부 로그 시스템                               |
| 시간 기록    | 직접 작성해야 함 | 자동 포함 가능                                        |
| 에러 추적    | 제한적       | `exc_info=True`로 스택 트레이스 기록 가능                  |
| 운영 환경 제어 | 어려움       | 환경별 로그 레벨 조정 가능                                 |
print()는 디버깅용 임시 출력에 가깝고, logging은 운영 환경에서 에러를 추적하고 관리하기 위한 표준 방식입니다.
